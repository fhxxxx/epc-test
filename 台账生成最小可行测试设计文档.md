# 台账生成最小可行测试设计文档（MVP）

## 1. 目的
在不接入数据库/数据湖API的前提下，基于本地 Excel 文件做一个最小 Demo，验证“多源取数 + 计算 + 新表生成”是否可行，并覆盖以下能力：
- 从多个本地 sheet 取数并计算生成新 sheet
- 同一 sheet 内公式计算
- 跨 sheet 取数与计算
- 样式与合并单元格保留/复制
- 公式重算后结果可校验（无 `#REF!`、`#DIV/0!`）

## 2. 输入资料与当前事实

### 2.1 参考设计文档（overview）关键要求
- 主流程要求“收集数据源 -> 生成各 sheet -> 组装 Excel -> 后置校验”。
- 后置校验包含：公式错误检查、样式/合并单元格不破坏。
- 公式策略是“混合模式”：系统计算字段写值，部分公式保留。
- 样式策略：以模板为唯一样式来源，运行时尽量只填值。

### 2.2 模板文件（税务台账生成逻辑_V2.0.xlsx）结构观察
基于本地模板解析（使用 `_tmp_v20.xlsx`）得到：
- 总 sheet 数：29
- 公式总数：2333
- 含跨 sheet 公式：130
- 含外部工作簿引用公式：130（如 `[112]xx`、`[115]xx`）

### 2.3 可行性风险（你怀疑点的核心）
1. **Aspose.Cells 依赖已补齐（已确认）**：`pom.xml` 已包含 `com.aspose:aspose-cells`。当前风险转为运行时验证（license、公式重算行为、样式兼容性）是否满足预期。
2. **模板存在外部工作簿引用**：若不替换/降级，公式重算可能失败或结果不稳定。
3. **sheet 名精确匹配风险**：例如存在 `Summary `（尾部空格），代码中按常量取 sheet 时容易取错。
4. **样式复制复杂**：若需要插入行，需明确复制样式与公式策略，否则模板结构容易破坏。

结论：先做“最小闭环 Demo”是必要且正确的，先验证核心链路，再扩展到完整台账。

## 3. MVP 范围（本次 Demo 只做这些）

### 3.1 In Scope
- 输入：单一模板文件（本地）
- 读取至少 3 个源 sheet 的数据
- 生成 1 个新 sheet：`Demo_台账输出`
- 新 sheet 内同时存在：
- 代码写入的值
- 同 sheet 公式
- 跨 sheet 公式
- 对关键区域应用样式复制（表头、金额列格式）
- 执行公式重算并输出结果文件
- 自动校验：无 `#REF!` / `#DIV/0!`

### 3.2 Out of Scope
- 不接数据库、不接 API
- 不做 26+ 全量 sheet 生成
- 不做权限、上传下载流程
- 不做跨月历史累积逻辑

## 4. Demo 测试对象设计

### 4.1 输入 sheet（建议固定）
- `PL附表-2320、2355`
- `增值税销项 `（注意尾部空格）
- `增值税进项认证清单`
- `公司代码配置表`（用于映射字段/税率示例）

### 4.2 目标输出 sheet
- 新增：`Demo_台账输出`

建议列结构：
1. `项目编码`（来自配置或源表）
2. `PL金额`（取源表值）
3. `销项税额`（取源表值）
4. `进项税额`（取源表值）
5. `应纳税额`（同 sheet 公式：`=C2-D2`）
6. `校验值`（跨 sheet 公式：引用 `Summary ` 或其它稳定 sheet 的单元格）
7. `状态`（代码写值：`OK/FAIL`）

## 5. 技术方案（建议）

### 5.1 推荐实现顺序
1. **V1（建议先做）**：直接使用 Aspose.Cells 实现 MVP（与设计文档一致）
2. **V2（可选）**：保留 Apache POI 兜底实现（用于对比验证或在 Aspose 受限场景下替代）

### 5.2 关键实现点
- 只从模板复制样式，不在代码硬编码颜色边框。
- 对“系统计算字段”优先写值；对“可解释汇总字段”保留简单公式。
- 外部工作簿引用公式统一策略：
- MVP 中不依赖外链公式作为断言依据
- 必要时将其回填为值后再重算
- sheet 名使用常量 + trim 比较 + 存在性校验，避免尾空格问题。

## 6. 用例矩阵（最小但完整）

| 用例ID | 目标 | 输入 | 断言 |
|---|---|---|---|
| TC-01 | 多 sheet 取数 | 3+ 源 sheet 有效数据 | `Demo_台账输出` 生成且行数>0 |
| TC-02 | 同 sheet 公式 | 输出 sheet `应纳税额` 使用 `=C-D` | 重算后数值正确 |
| TC-03 | 跨 sheet 计算 | 输出 sheet 引用 `Summary ` 某单元格 | 公式存在且重算不报错 |
| TC-04 | 样式复制 | 表头/金额列复制模板样式 | 字体、填充、数字格式与模板一致 |
| TC-05 | 合并单元格保留 | 输出表头设置合并 | 合并区域存在且未破坏 |
| TC-06 | 错误兜底 | 源 sheet 缺失/名称不匹配 | 明确报错（包含缺失 sheet 名） |
| TC-07 | 公式异常检查 | 人工注入错误公式 | 能识别 `#REF!/#DIV/0!` 并 fail |

## 7. 验收标准（Go/No-Go）

### 7.1 Go（通过）
- 能生成 `Demo_台账输出` 且包含预期列
- 通过至少 7 个用例中的 6 个，且 TC-01/02/03/04 必须通过
- 输出文件中无 `#REF!`、`#DIV/0!`
- 样式比对通过（至少表头 + 金额列）

### 7.2 No-Go（不通过）
- 无法稳定重算公式
- 跨 sheet 取数不稳定
- 样式明显破坏（颜色、格式、合并错误）

## 8. 建议代码结构（Demo）

```text
src/main/java/.../ledgerdemo/
  LedgerDemoRunner.java          // main入口
  TemplateLoader.java            // 加载模板与sheet定位

  FormulaValidator.java          // 公式重算与错误检查
  DemoResultAssert.java          // 断言输出
```

## 9. 关键断言细化

- 文件级断言：输出文件存在、可打开、sheet 数量正确
- 结构断言：`Demo_台账输出` 列头、行数、合并单元格范围
- 公式断言：
- 指定单元格 `CellType=FORMULA`
- 公式字符串符合预期（如 `C2-D2`、`'Summary '!B10`）
- 重算后 `CachedFormulaResultType` 合法
- 值断言：随机抽 3~5 行做精确数值比对（含四舍五入规则）
- 样式断言：对比 `styleIndex` 或关键属性（字体、填充、数据格式）
